¿Qué es?

Datalake analítico para una comercializadora de energía.

La empresa compra electricidad a generadores en el mercado mayorista (eólica, hidroeléctrica, nuclear, solar, biomasa) y la revende a clientes residenciales, comerciales e industriales. Su sistema transaccional exporta proveedores, clientes y transacciones que necesitan ser ingestados, transformados y analizados a escala.

¿Para qué sirve esta solución?

Convierte CSV operacional en analítica gobernada.

El pipeline ingiere CSVs particionados por fecha de carga, los cataloga automáticamente, los transforma a Parquet+Snappy en una zona procesada, expone consultas SQL desde Python con Athena, y opcionalmente carga el resultado a un datawarehouse Redshift. Lake Formation gobierna el acceso por dominio (PII vs Public) sin escribir bucket policies a mano.

¿Para quién es esta entrega?

Una prueba técnica de Data Engineer Senior+ en AWS.

El evaluador puede recorrer las pestañas (1 por bloque del enunciado), ejecutar queries Athena en vivo, descargar el PDF autocontenido con todas las respuestas, ver el dashboard del Ej.2 con datos reales de 90.000 ticks, y verificar el costo medido en tiempo real desde las APIs de AWS.

    R1

    Estrategia datalake S3 · capas + particionado por fecha de carga.

    1
    Qué pide la prueba
    Crear una estrategia de datalake en s3 con las capas que usted considere necesario tener y cargue esta información de manera automática y periódica. Los archivos deben particionarse por fecha de carga.
    PruebaAWS.pdf · Ejercicio 1, requisito técnico 1
    2
    Mi enfoque

    Diseñé un datalake de 6 capas en S3 (no 3) para separar "datos analíticos internos" de "artefactos publicados al usuario" y de "área temporal de cómputo". Cada capa tiene un propósito único, una bucket policy distinta y una política de lifecycle propia. Para el particionamiento por fecha de carga uso Hive-style (loaded_at=YYYY-MM-DDTHH/) que es el formato que Athena, Glue y Redshift Spectrum entienden nativamente.

    La carga periódica la programo desde Glue Crawler (cron horario) y, para producción, sería un EventBridge schedule disparando el extractor Python.

    3
    Servicios y técnicas que apliqué (con contexto)

    Usé S3 como almacenamiento (la base del datalake) y la convención de particionado Hive-style para que las queries downstream filtren eficientemente sin escanear todo. Las dos piezas son complementarias: S3 me da durabilidad y bajo costo; el particionado me da eficiencia analítica.

    Amazon S3 Simple Storage Service
    Qué es
    Servicio de almacenamiento de objetos serverless: cualquier archivo (CSV, Parquet, JSON, imágenes...) accesible vía HTTP/HTTPS. Pagas por GB almacenado y por requests. Durabilidad 99.999999999%.
    Rol aquí
    Columna vertebral del datalake. 6 buckets con propósitos separados (raw, staging, processed, public, athena-results, glue-scripts). SSE-S3 obligatoria; block public access excepto el bucket "public".
    Particionamiento Hive-style key=value/
    Qué es
    Convención de nombrado donde cada subdirectorio es columna=valor/. Athena, Glue, Spark y Redshift Spectrum la entienden nativamente para podar (skip) datos al hacer queries con filtros.
    Rol aquí
    Cada extracción crea loaded_at=YYYY-MM-DDTHH/. Permite reprocesar una sola hora sin afectar el histórico, y filtrar en SQL WHERE loaded_at > '2026-04-15' escanea solo lo nuevo.
    4
    Implementación concreta

    Las 6 capas que creé

    • s3://prueba-aws-raw/ · CSVs originales extraídos del RDS · zona "landing"
    • s3://prueba-aws-staging/ · output intermedio de Glue (temp, scratch para Spark)
    • s3://prueba-aws-processed/ · Parquet+Snappy particionado · zona "curated"
    • s3://prueba-aws-public/ · assets descargables (PDF, reportes calidad, JSONs Athena)
    • s3://prueba-aws-athena-results/ · output queries Athena · lifecycle 7 días
    • s3://prueba-aws-glue-scripts/ · código PySpark de los 3 jobs ETL

    ¿Por qué 6 capas y no 3?

    Separo "publicación pública" (PDFs, reportes) de "datalake interno" para que Lake Formation gobierne uno y bucket policies abierto el otro, sin mezclar permisos. Athena results en bucket separado evita confusión con el datalake principal y permite lifecycle agresivo (queries son efímeras).

    El particionado por fecha de carga

    El extractor Python (scripts/02_extract_to_s3_raw.py) crea una partición nueva por cada corrida con granularidad horaria:

    s3://prueba-aws-raw/<tabla>/loaded_at=YYYY-MM-DDTHH/<tabla>.csv

    Beneficios prácticos:

    • Idempotencia: re-correr no contamina histórico (cada run = nueva partición)
    • Reproceso quirúrgico: borrar una partición específica si vino corrupta sin afectar otras
    • Filtrado eficiente en Athena: WHERE loaded_at > '2026-04-15' escanea solo lo nuevo
    • Trazabilidad: loaded_at es metadata útil para auditoría y debugging

    3 transformaciones Glue · Parquet en zona procesada.

    1
    Qué pide la prueba
    Realice 3 transformaciones básicas de datos utilizando AWS Glue y transforme la información para que esta sea almacenada en formato parquet en una zona procesada.
    PruebaAWS.pdf · Ejercicio 1, requisito técnico 2
    2
    Mi enfoque

    En vez de meter las 3 transformaciones en un solo job monolítico, las separé en 3 jobs independientes con responsabilidades distintas: uno para dimensiones (SCD-1), uno para el hecho transaccional (con derivaciones y particionado por año/mes), y uno para pre-agregados mensuales. Esta separación me permite ejecutarlos en paralelo, debuggearlos por separado y reentrenar uno solo si cambia la lógica.

    Los 3 jobs son PySpark sobre Glue 4.0 con worker G.1X (1 DPU = 4 vCPU + 16 GB) × 2 workers, suficiente para los volúmenes del ejercicio. Cada uno escribe Parquet+Snappy en una sub-carpeta de la zona processed.

    3
    Servicios y formatos que apliqué (con contexto)

    Glue es el motor de ejecución (Spark gestionado), PySpark el lenguaje en que escribo las transformaciones, Parquet el formato de salida (columnar, optimizado para queries analíticas) y Snappy el codec de compresión que balancea ratio + velocidad de lectura. Las cuatro piezas son la combinación canónica para datalakes analíticos modernos en AWS.

    AWS Glue (ETL) serverless ETL
    Qué es
    Servicio serverless para ejecutar jobs ETL sobre Apache Spark. AWS gestiona el cluster Spark, tú solo escribes código PySpark / Scala. Facturación por DPU-hora (Data Processing Unit, equivalente a 4 vCPU + 16 GB).
    Rol aquí
    Corre los 3 jobs PySpark que transforman raw → processed. Glue 4.0 con worker G.1X (1 DPU por worker) y 2 workers por job. Cada run dura ~50-70 segundos y consume ~0.03-0.04 DPU-hr.
    Apache Parquet columnar
    Qué es
    Formato de archivo columnar (no por filas como CSV). Comprime mejor (ratio 3-10×), escanea solo las columnas que la query necesita, y es estándar de facto para datalakes analíticos.
    Rol aquí
    Output de los 3 ETL jobs en S3 processed. Athena y Redshift leen Parquet nativamente; reduce escaneo en Athena (que cobra por TB escaneado) entre 80% y 95% vs CSV.
    Snappy compression codec
    Qué es
    Codec de compresión rápido (no el que más comprime, pero sí el más balanceado para Spark). Permite split de archivos sin descomprimir todo (paralelización Spark).
    Rol aquí
    Codec aplicado a los Parquet del processed. Recomendado por defecto en Glue + Athena (gzip comprime más pero no permite split eficiente).
    PySpark Spark API en Python
    Qué es
    API Python para Apache Spark. Permite definir transformaciones distribuidas con DataFrames y SQL. Glue añade DynamicFrame con resolución automática de tipos heterogéneos.
    Rol aquí
    Lenguaje de los 3 jobs ETL. Cada job lee CSV de raw, aplica transformaciones (SCD-1, joins, agregados), escribe Parquet con particionamiento.
    4
    Las 3 transformaciones que implementé

    Cada tarjeta es un job de Glue con su propósito de negocio, su input desde S3 raw y su output en una sub-carpeta de S3 processed. Los 3 los orquesto desde scripts/04_run_glue_etl.py en orden secuencial (pero corren en Glue independientemente).

    Glue Crawler · catálogo automático de esquemas.

    1
    Qué pide la prueba
    Utilizando AWS Glue, crea un proceso que detecte y catalogue automáticamente los esquemas de los datos almacenados en el datalake.
    PruebaAWS.pdf · Ejercicio 1, requisito técnico 3
    2
    Mi enfoque

    Configuré 2 crawlers separados: uno escanea la zona raw (CSVs) en horario automático cada hora, y otro escanea la zona processed (Parquet) on-demand cuando termina un ETL. Las tablas detectadas se registran en el Glue Data Catalog, que es el metastore unificado que comparten Athena, Redshift Spectrum, EMR y Lake Formation, evitando definir schemas múltiples veces.

    El crawler detecta automáticamente las particiones loaded_at=... creadas por el extractor (R1), sin escribir DDL manual.

    3
    Servicios que apliqué (con contexto)

    Glue Crawler es la herramienta de detección; Glue Data Catalog es el "diccionario" donde quedan las tablas registradas; y el cron scheduler de Glue es lo que automatiza la periodicidad. Las tres piezas trabajan juntas para cumplir literalmente el requisito de "automáticamente".

    Glue Crawler schema discovery
    Qué es
    Servicio que escanea S3 (o JDBC, DynamoDB, Mongo) y deduce el esquema automáticamente: detecta columnas, tipos, particiones. Crea o actualiza tablas en el Glue Data Catalog.
    Rol aquí
    2 crawlers: raw (CSV horario) y processed (Parquet on-demand). Detectan particiones loaded_at=... automáticamente; sin DDL manual.
    Glue Data Catalog metastore
    Qué es
    Repositorio centralizado de metadatos (databases + tables + columns + locations). Compatible con Apache Hive metastore. Fuente única de schema para Athena, Redshift Spectrum, EMR y Lake Formation.
    Rol aquí
    Database datalake_energia con 11 tablas catalogadas (5 raw + 6 processed). Athena, Redshift y EC2 backend la consultan vía AWS SDK.
    Cron Glue scheduler automation
    Qué es
    Glue permite programar crawlers y jobs con cron expressions estilo AWS (diferente al cron Unix: incluye campo de día-de-semana con ?).
    Rol aquí
    0 * * * ? * = cada hora a la hora exacta (minuto 0). Crawler raw escanea cada hora para detectar nuevas particiones. Crawler processed corre on-demand.
    4
    Configuración concreta

    Los 2 crawlers que configuré

    • prueba-aws-crawler-raw · escanea raw cada hora a la hora exacta (cron 0 * * * ? * en sintaxis Glue: minuto 0, todas las horas, todos los días/meses, cualquier día de semana)
    • prueba-aws-crawler-processed · on-demand · registra Parquet generados por los jobs ETL

    ¿Por qué cada hora y no cada minuto?

    El sistema fuente exporta CSVs en lotes (no streaming). Una corrida horaria detecta nuevas particiones loaded_at=YYYY-MM-DDTHH/ sin gastar DPU innecesario. Si se requiere baja latencia, lo cambio a EventBridge → Lambda → StartCrawler disparado por s3:ObjectCreated:Put.

    R5

    Esquemas separados por dominio · Ej.1 vs Ej.2.

    1
    Por qué incluyo este punto (no es requisito explícito)
    La prueba tiene dos ejercicios prácticos (energía y divisas) y comparten el mismo cluster RDS por ahorro de costo.
    Decisión arquitectónica propia · explicada para el evaluador
    2
    Mi enfoque

    Aunque ambos ejercicios viven en el mismo cluster RDS por economía, las tablas viven en schemas separados por dominio: core+dwh+audit son de energía (Ej.1), divisas es de compra/venta dólares (Ej.2). Esto sigue el principio de Bounded Context de Domain-Driven Design: cada dominio tiene su propio vocabulario, modelo y permisos, sin colisiones de nombres ni mezcla accidental de tablas.

    El resultado: si mañana migro Ej.2 a su propio cluster, la separación ya está hecha; solo cambia el host de conexión.

    3
    Conceptos que apliqué (con contexto)

    Bounded Context es la teoría (DDD); PostgreSQL Schemas es el mecanismo técnico que lo implementa nativamente sin necesidad de bases de datos separadas.

    Bounded Context DDD
    Qué es
    Concepto de Domain-Driven Design: cada dominio del negocio tiene su propio modelo, vocabulario y schema. Evita mezclar conceptos de dominios distintos en la misma tabla.
    Rol aquí
    Ej.1 (energía) y Ej.2 (divisas) viven en schemas separados aunque compartan el mismo cluster RDS. Esto facilita evolucionar cada uno independientemente.
    PostgreSQL Schemas namespace
    Qué es
    Mecanismo nativo de Postgres para namespacing de tablas. Una base puede tener N schemas, cada uno con sus propias tablas, vistas, funciones, sin colisiones.
    Rol aquí
    4 schemas: core, dwh, audit (Ej.1) y divisas (Ej.2). Permisos por schema vía GRANT USAGE ON SCHEMA.

    Por qué Terraform y no CloudFormation/CDK

    Elegí Terraform por tres razones:

    • Multi-cloud: si mañana hay que migrar parcialmente a GCP/Azure, los módulos de network/storage se adaptan. CloudFormation queda atado a AWS.
    • State explícito: terraform state list da visibilidad inmediata de los recursos creados. CloudFormation oculta esto detrás de Stacks que son menos transparentes.
    • Comunidad y módulos: el registro de Terraform Hub tiene módulos battle-tested para todos los servicios AWS (terraform-aws-modules/*) que ahorran semanas de desarrollo.

    1
    Qué pide la prueba (puntos adicionales)
    Configure AWS Lakeformation para centralizar el gobierno, la seguridad y compartir los datos alojados en el datalake creado.
    PruebaAWS.pdf · Ejercicio 1, puntos adicionales 2
    2
    Mi enfoque

    Configuré Lake Formation con permisos por tags (LF-Tags), no por bucket policies de S3. Esto centraliza el gobierno: marcas la tabla como Sensitivity=PII una sola vez, y todos los principals que no tienen permission sobre ese tag quedan bloqueados automáticamente. Adicionalmente registré s3://prueba-aws-processed/ como ubicación gobernada por LF para que las bucket policies dejen de ser la fuente principal de control.

    El resultado: el rol del backend EC2 puede leer tablas con Sensitivity ∈ {Public, Internal} pero no las marcadas como PII, sin escribir una sola línea de bucket policy.

    3
    Servicios que apliqué (con contexto)

    Lake Formation es la capa de gobierno; LF-Tags es el mecanismo declarativo de access control; IAM sigue ahí pero como capa base (LF se evalúa DESPUÉS de IAM).

    AWS Lake Formation data lake governance
    Qué es
    Capa de gobierno encima de S3 + Glue Catalog. Centraliza permisos finos (table, column, row) usando LF-Tags reutilizables, sin escribir bucket policies a mano. Auditoría vía CloudTrail.
    Rol aquí
    Marca s3://prueba-aws-processed/ como ubicación gobernada. Define LF-Tags Sensitivity y Domain. Concede SELECT al backend EC2 solo para tablas con Sensitivity ∈ {Public, Internal}.
    LF-Tags tag-based access control
    Qué es
    Atributos clave-valor (ej: Sensitivity=PII) asignables a databases, tablas y columnas. Las policies se definen sobre tags, no sobre nombres de recursos: escala mejor.
    Rol aquí
    2 LF-Tags definidos: Sensitivity (Public/Internal/PII) y Domain (Provider/Client/Transaction/Reference). Marcas la tabla una vez, los permisos se aplican automáticamente.
    AWS IAM Identity and Access Management
    Qué es
    Sistema de identidades, roles y políticas de AWS. Sin costo. Define quién puede hacer qué sobre qué recursos. Lake Formation se evalúa DESPUÉS de IAM (dos capas).
    Rol aquí
    4 roles separados con least privilege: Glue, EC2 backend, Redshift, Lake Formation service. Sin *:*; permission boundaries para limitar blast radius.
    4
    Configuración aplicada y por qué LF gana a IAM puro

    Configuración aplicada

    • Data Lake Admins: el ARN del usuario IAM aplicador de Terraform · revocable sin tocar bucket policies
    • Service role: prueba-aws-lakeformation-service que LF asume para acceder a S3 governed locations
    • Data lake location registrada: s3://prueba-aws-processed/ queda bajo gobierno LF (las bucket policies sobre este bucket dejan de ser la fuente de verdad)
    • LF-Tags definidos:
      • Sensitivity → valores Public · Internal · PII
      • Domain → valores Provider · Client · Transaction · Reference
    • Permisos fine-grained por tag (no por bucket policy directa):
    • Backend EC2: SELECT, DESCRIBE sobre tablas con Sensitivity ∈ {Public, Internal}
    • Backend EC2 NO ve tablas con Sensitivity = PII
    • Glue service-role: DATA_LOCATION_ACCESS sobre processed para escribir Parquet
    • Redshift role: SELECT sobre las dim/fact para hacer COPY

    Por qué Lake Formation y no IAM puro

    Con IAM tradicional, dar acceso a "tablas no-PII" del datalake requiere policies con condiciones Resource por bucket-prefix que se vuelven inmanejables conforme crecen los datasets. Lake Formation centraliza el modelo en LF-Tags reusables: marcas la tabla como Sensitivity=PII una vez, y todos los principals que NO tienen permission sobre ese tag quedan automáticamente excluidos.

    Esto habilita patrones imposibles con IAM puro:

    • Column-level security · solo ciertos roles ven la columna id_externo de cliente; otros ven la fila pero esa columna aparece como NULL
    • Row-level security · usuarios de Bogotá solo ven transacciones de Bogotá vía Data Filter con expresión SQL
    • Cross-account sharing · compartir tablas a otra cuenta AWS sin replicar S3 (la cuenta destino consulta las mismas Parquet via cross-account grant)
    • Data Lake Permissions vs IAM · LF tiene su propio modelo de evaluación que se aplica DESPUÉS de IAM; un usuario puede tener IAM s3:GetObject pero LF lo bloquea por tag

    Workflow operativo

    1. Data engineer marca la nueva tabla con LF-Tag al crearla en Glue Catalog
    2. Data steward (rol separado) define qué LF-Tags puede tocar cada principal
    3. Auditoría revisa lakeformation:GrantPermissions events en CloudTrail
    4. Cero coordinación cross-team para policies de bucket

    Mi enfoque

    Elegí Redshift Serverless sobre Provisioned porque para un demo con uso esporádico el modelo serverless cuesta $0 cuando está idle (auto-pause a los 60s) y solo cobra ~$0.36/RPU-hora cuando hay queries activas.     Servicios y técnicas que apliqué (con contexto)

    Redshift Serverless es el datawarehouse; COPY es el comando de carga masiva de Redshift; Redshift Data API es el SDK serverless para ejecutar SQL sin manejar conexiones JDBC; y DISTKEY+SORTKEY+ENCODE son las 3 optimizaciones clave del modelo dimensional en Redshift.

    Amazon Redshift Serverless data warehouse
    Qué es
    Versión serverless de Redshift (DW columnar AWS). Sin cluster que provisionar: defines un namespace + workgroup, pagas por RPU-hora consumida (Redshift Processing Unit). Auto-pause tras inactividad.
    Rol aquí
    Namespace prueba-aws-ns + workgroup base 8 RPU. Carga el fact con COPY desde Parquet processed. Auto-pause a los 60s · costo idle $0.
    COPY (Redshift) bulk load
    Qué es
    Comando masivo de carga de Redshift que paraleliza la ingesta automáticamente por slice. Soporta CSV, JSON, Parquet, ORC, AVRO directo desde S3 con un IAM Role.
    Rol aquí
    COPY analytics.fact_transaccion FROM 's3://.../fact/' IAM_ROLE 'prueba-aws-redshift-role' FORMAT PARQUET. Carga 10K filas en segundos.
    Redshift Data API async SQL
    Qué es
    API REST/SDK para ejecutar SQL en Redshift sin manejar conexiones JDBC persistentes. Asíncrono: execute_statement → describe_statement → get_statement_result.
    Rol aquí
    Script 06_load_to_redshift.py usa boto3 redshift-data client. Sin pool de conexiones que gestionar; ideal para Lambda y workloads esporádicos.
    DISTKEY + SORTKEY + ENCODE columnar tuning
    Qué es
    Tres optimizaciones clave en Redshift: distribución por clave (joins en mismo nodo), sort interno (range queries), compresión por columna (auto-elegida).
    Rol aquí
    Recomendado: DISTKEY(cliente_id) · SORTKEY(fecha_id, tipo_energia_id) · ENCODE AUTO. Optimiza joins con dim_cliente y queries por rango temporal.

    Diseño técnico, script y por qué Serverless vs Provisioned

    Diseño técnico

    • Engine: Redshift Serverless · auto-pause · auto-scale en RPUs (Redshift Processing Units)
    • Namespace: prueba-aws-ns con admin user, KMS-managed encryption por defecto
    • Workgroup: prueba-aws-wg base 8 RPU · max 8 RPU (mínimo absoluto que Redshift permite)
    • Schema destino: analytics.fact_transaccion · mismo modelo que dwh.fact en RDS para reconciliación trivial
    • Carga masiva: COPY ... FROM 's3://prueba-aws-processed/fact/' IAM_ROLE 'prueba-aws-redshift-role' FORMAT PARQUET
    • Validación post-COPY: query de conteo SELECT COUNT(*), SUM(monto_usd), MIN(fecha_id), MAX(fecha_id) para reconciliación con S3 source
    • Pause automático: workgroup entra en idle tras 60 segundos sin queries · siguiente query reactiva en menos de 2 segundos

    Estrategia de carga

    Tres opciones evaluadas:

    1. COPY directo desde S3 Parquet (elegido) · sin transformación intermedia, máxima velocidad, paralelización automática por slice
    2. Redshift Spectrum · query S3 sin copiar · costo por TB escaneado, latencia más alta, pero $0 storage
    3. Glue ETL job dedicado · si necesitamos transformación adicional antes de cargar (no es el caso)

    Script y orquestación

    python scripts/06_load_to_redshift.py
# 1. CREATE SCHEMA IF NOT EXISTS analytics
# 2. CREATE TABLE IF NOT EXISTS analytics.fact_transaccion
# 3. TRUNCATE analytics.fact_transaccion
# 4. COPY FROM 's3://.../fact/'
#    IAM_ROLE 'arn:aws:iam::...:role/prueba-aws-redshift-role'
#    FORMAT PARQUET
# 5. SELECT validation (count, sum, min/max date)
# 6. Workgroup queda en idle (auto-pause)

    Por qué Serverless y no Provisioned

    Para volumen pequeño-medio con uso esporádico (analítica nocturna, dashboards diarios), Serverless paga solo cuando hay queries activas:

    • Costo idle: $0/hr (workgroup pausado)
    • Costo activo: ~$0.36 por RPU-hora · facturación por segundo (mínimo 60s)
    • Cold start: 30-60s la primera query del día (workgroup wake-up); siguientes en menos de 100ms

    Para uso 24×7 sostenido por encima de 8 RPU continuos, RA3 provisioned con reserved instances 1-año saldría ~50% más barato. La heurística que aplico:

    • Usage rate mayor a 60% del tiempo → Provisioned + Reserved Instance
    • Usage rate menor a 60% o esporádico → Serverless
    • Cargas batch nocturnas predecibles → Serverless con auto-pause
    • Dashboards 24×7 con concurrencia alta → Provisioned + Concurrency Scaling

    Modelo dimensional en Redshift

    • Distribución: DISTKEY(cliente_id) · joins con dim_cliente quedan en mismo nodo
    • Sort key: SORTKEY(fecha_id, tipo_energia_id) · queries por rango temporal optimizadas
    • Compresión: ENCODE AUTO · Redshift elige el mejor encoding por columna basándose en sample
